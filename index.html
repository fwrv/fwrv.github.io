Converting this site's content into JS objects and eventually JSON files

(Talking the talk on 12 Feb 2016, walking the walk soon) The "great leap forward" for this site will be when content can be displayed / modified in a modular and scalable way. Right now, the site you are looking at is one long html file. (If you're using a chrome browser, you can right-click/control-click on the screen, hit "Inspect", and see the code for yourself.) Each update that I make, I type into the same file. This is increasingly untenable as the number of updates grows - most importantly, it's inelegant!

As a key part of my javascript education, I've been learing about objects. Within the context of js, an object is a "complex" data type, as opposed to numbers or strings of text, which are "simple" data types. Objects are useful in that they can take on multiple properties. For example, a smartphone-contact object could contain the properties of a person's name, their number and their email address. Curating an array (or list) of objects all with similar (or overlapping) properties allows you to list or search within quite easily (using "for" loops, among other nifty tricks).

The next step, then, for this site/project is to convert the content posts into objects, each with a set of properties. The common properties as I've laid them out include:

  • Title
  • Date posted
  • Tag (i.e. whether it counts as "making", "reading", or "eating")
  • Summary (a one sentence description which can be displayed in an index or TOC)
  • Author (for a future scenario in which the BLC may contain kind outside contributors)
  • Content (i.e. the post itself)

When the content lives in the form of js objects, it will free the index.html page (and other html sub-pages, as I make them) from housing all the content in the same doc. You, the viewer, will be able to search and sort content by tag, date or author or even browse by summary. The entire content process will become more modular and I'll be able to abstract fundamental app changes without getting ensnared by a long thorny html file. It is the future and it is exciting.

However, this can be improved upon still further! You may notice a shortcoming of the plan so far: By turning content posts into js objects in a single array I may be taking them out of the html file, but I'm only just moving them all into a js one. This is where JSON comes in. JSON, or "JavaScript Object Notation" is a format and file type that will allow me to create content posts as independant files with the same data parameters that I would feed into the objects in my standard js file. I haven't fully gotten my head around this yet - but the idea is that each of the content posts would live as its own file in my github repository and my main BLscript.js file would reference them as described above.

These changes will bring about a brave new world for the members of the Big Lake Club. I'll be providing unsolicited updates as I go along.

(12 Feb 2016): "Business Models, Information Technology, and the Company of the Future" by Stanford's Haim Mendelson showed up this morning in the daily newsletter from the MIT tech review (the piece was originally published in Oct 2015). The heady title gives a sense of what to expect from the piece, which is long on MBA-lingo and short on real-world examples / applications. (Seriously though, this is an actual quote: "The combined effect of mobile technologies, wearable devices and sensors, cloud computing, and big data technologies will sharpen the structure of future business models." Say what? No way!) That being said (and as pejorative as I made "MBA-lingo" sound just now), I found the following interesting:

Three technologies will play a particularly important role in the use of IT:

  • Customer-intimacy agents will be customers digital representatives in the marketplace, using data to find and solicit solutions that will make them better off.
  • Value-chain coordinators will match supply and demand, assemble customized solutions, and engage in electronic customer data-driven innovation.
  • Producers will increasingly engage in marketplace selling and data-driven innovation.

These three technologies seem to fit along a continuum of being customer facing vs. enterprise facing. "Customer-intimacy agents" purport to be actively champion-ing (even anticipating) the needs/desires of the customer/user. An example of this could be (my employer) Knewton's adaptive learning technology, which uses student proficiency data to cater content to the individual learner's needs and goals (and showing teachers predictive analytics re: student performance).

"Value-chain coordinators" straddle the best interests of the customer and the enterprise, in theory connecting the one with the best possible services of the other. An obvious example of this would be how Uber uses surge pricing to match ride supply with ride demand. Surge pricing itself is unsustainably punitive and really a manifestation of equilibrium breaking down rather than a triumph of efficiency - it's more of a stop-gap. The victory for "value-chain coordinating" technology will be when Uber can anticipate ride demand so well so that surge pricing wont even take place. Maybe you could debate that Uber would never want its own technology to make surge pricing obsolete but I think users will ultimately be powerful enough that it (Uber) won't have a choice (the power of Uber's users is also why European regulators and cab drivers are fighting a losing battle). This anticipatory tech would be far easier to implement when cars go driver-less. (For another example: see how Amazon is stocking it's "fulfillment centers" similarly to anticipate customer/user demand.

Mendelson's last category is also probably the most boring as it's already the most prevalant. Producers are selling into an increasingly flat market (see the post on ubiquitous/accessible content above) where intimate knowledge of their end users (and use cases) is imperative both for differentiation and innovation.

For all the drivel (Mendelson's and mine), I think there is actually good food for thought here.
